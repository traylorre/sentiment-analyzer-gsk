## Summary

Fix deterministic timing bug in `chaos-accessibility.spec.ts` where `toBeVisible()` succeeds but `AxeBuilder.analyze()` runs before the browser's accessibility tree has stabilized. Create a shared `waitForAccessibilityTree()` helper that uses `page.waitForFunction()` to poll for ARIA attributes, then apply it in all 3 failing tests between visibility checks and axe-core scans.

## Phase 0: Research

### Research Task 1: Accessibility Tree Update Timing

**Question**: How long does browser accessibility tree stabilization take after DOM mutation, and what is the recommended pattern for waiting?

**Finding**: The browser's accessibility tree is updated asynchronously after DOM mutations. The exact timing varies by browser engine:
- **Chromium**: Accessibility tree updates are batched and fire after the next rendering frame (typically 16ms at 60fps, but can be delayed under load)
- **Firefox**: Accessibility tree updates are synchronous with layout but asynchronous with the DOM mutation itself
- **WebKit**: Similar to Chromium but with slightly different batching behavior

**Best practice**: The recommended pattern is NOT to wait for a fixed time, but to poll for the specific accessibility properties that axe-core will check. This is because:
1. `page.waitForFunction()` runs in the page context and can check `element.getAttribute('aria-*')` or `element.textContent`
2. By the time the function finds the attributes present, the accessibility tree has also been updated (since attribute presence in DOM triggers accessibility tree recalculation)
3. This is the pattern used by the Playwright team internally for accessibility testing

**Decision**: Use `page.waitForFunction()` with a serialized check function that verifies specific DOM attributes are present and non-empty. This is reliable because the accessibility tree update is triggered BY the DOM attribute presence -- once we confirm the attributes are in the DOM and computed, the accessibility tree will have been updated.

### Research Task 2: `page.waitForFunction()` Cross-Browser Reliability

**Question**: Does `page.waitForFunction()` behave consistently across all 5 Playwright browser projects?

**Finding**: Yes. `page.waitForFunction()` is a core Playwright API that works identically across Chromium, Firefox, and WebKit. It polls using `requestAnimationFrame` by default, which means it checks once per rendering frame. This is ideal for accessibility tree waiting because:
1. It aligns with the rendering pipeline (DOM -> Style -> Layout -> Accessibility Tree -> Paint)
2. It does not busy-wait (no CPU waste)
3. The polling interval is automatically tuned to the browser's frame rate

**Decision**: Use `page.waitForFunction()` without custom polling interval -- rely on default `requestAnimationFrame`-based polling.

### Research Task 3: Axe-core and Accessibility Tree Dependency

**Question**: Does axe-core (`AxeBuilder.analyze()`) query the DOM or the accessibility tree?

**Finding**: Axe-core queries both:
1. **DOM-based checks**: Looks at raw DOM attributes (`role`, `aria-*`, `tabindex`)
2. **Computed accessibility checks**: Uses `window.getComputedStyle()` for visibility, `element.offsetWidth`/`offsetHeight` for rendered size, and the browser's accessibility APIs where available
3. **Accessible name computation**: Follows the W3C Accessible Name computation algorithm, which traverses the DOM tree to resolve `aria-labelledby`, `aria-label`, visible text content, and `title` attributes

The race condition occurs because axe-core runs its accessible name computation before the browser has finished computing accessible names for dynamically inserted elements. Specifically:
- React inserts the element into the DOM (visible to `toBeVisible()`)
- Axe-core starts scanning and begins accessible name computation
- The browser has not yet finished computing accessible names for all elements (especially buttons with icon + text children)
- Axe-core reports violations like "button has no accessible name" which are transient

**Decision**: Wait for specific text content and ARIA attributes to be present in the DOM before running axe-core. Once these are in the DOM, axe-core's accessible name computation will find them.

## Phase 1: Design

### Data Model (Types)

```typescript
/**
 * Represents an expectation for an element's accessibility attributes.
 * Used by waitForAccessibilityTree() to know what to wait for.
 */
interface A11yExpectation {
  /** CSS selector for the element to check */
  selector: string;
  /**
   * ARIA/accessibility attributes that must be present and non-empty.
   * Special values:
   * - 'textContent': element must have non-empty textContent
   * - Any other string: treated as an attribute name (e.g., 'aria-live', 'aria-label')
   */
  attributes: string[];
}

/**
 * Options for waitForAccessibilityTree().
 */
interface WaitForA11yOptions {
  /** Maximum time to wait in milliseconds. Default: 5000 */
  timeout?: number;
}
```

### Contract: `waitForAccessibilityTree()`

**Input**:
- `page: Page` -- Playwright page object
- `expectations: A11yExpectation[]` -- list of elements and their expected attributes
- `options?: WaitForA11yOptions` -- optional configuration

**Output**: `Promise<void>` -- resolves when all expectations are met, rejects on timeout

**Behavior**:
1. Serialize the expectations into a check function
2. Pass to `page.waitForFunction()` which runs in the browser context
3. The check function iterates all expectations:
   - For each expectation, `document.querySelector(selector)` must find an element
   - For each attribute in `attributes`:
     - If `'textContent'`: `element.textContent.trim()` must be non-empty
     - Otherwise: `element.getAttribute(attribute)` must be non-null and non-empty
4. Return `true` only when ALL expectations pass
5. On timeout, throw with a message listing which expectations failed

### Usage Pattern (per-test)

```typescript
// T025: Health banner
await expect(banner).toBeVisible();
await waitForAccessibilityTree(page, [
  { selector: '[role="alert"]', attributes: ['aria-live'] },
  { selector: '[role="alert"] button', attributes: ['aria-label'] },
]);
const results = await new AxeBuilder({ page }).withTags([...]).analyze();

// T026: Error boundary fallback
await expect(page.getByText(/something went wrong/i)).toBeVisible();
await waitForAccessibilityTree(page, [
  { selector: 'button', attributes: ['textContent'] },
]);
const results = await new AxeBuilder({ page }).withTags([...]).analyze();

// T027: Error boundary buttons
await expect(page.getByText(/something went wrong/i)).toBeVisible();
await waitForAccessibilityTree(page, [
  { selector: 'button', attributes: ['textContent'] },
]);
// proceed with getByRole and focus assertions
```
