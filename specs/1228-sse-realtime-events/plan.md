# Implementation Plan: Wire SSE Real-Time Events

**Branch**: `1228-sse-realtime-events` | **Date**: 2026-03-20 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/1228-sse-realtime-events/spec.md`

## Summary

Wire the dead code in the SSE Lambda (`_create_sentiment_event()` and `_create_partial_bucket_event()`) to live data by:
1. Extending `PollingService` to compute per-ticker aggregate sentiment and detect changes between polls (FR-001/FR-003)
2. Adding a timeseries polling path against TIMESERIES_TABLE for bucket change detection (FR-002/FR-004)
3. Wiring both event types into `generate_global_stream()` and `generate_config_stream()` with ticker filtering (FR-006)
4. Fixing pre-existing bug: `polling.py` uses `item.get("ticker")` but items have `matched_tickers` (list)

## Key Technical Decisions

### D1: Per-Ticker Aggregate Change Detection (FR-001/FR-003)

**Approach**: Extend `PollingService.poll()` to return per-ticker aggregates alongside global metrics. Compute per-ticker score/label/count in the same `_aggregate_metrics()` pass from items already fetched via `by_sentiment` GSI.

**Change detection is per-connection**: The `PollingService` returns raw data (current metrics + per-ticker aggregates). Each stream generator coroutine maintains its own local snapshot and diffs against it. This prevents the shared-state race where connection A's poll updates the snapshot before connection B sees the change.

**Why**: The existing polling already queries all sentiment items. Computing per-ticker aggregates is O(n) in the same pass — no additional DynamoDB queries needed.

**Bug fix**: Change `item.get("ticker")` to iterate `item.get("matched_tickers", [])` so `by_tag` is correctly populated. Each item can match multiple tickers (e.g., an article about "Apple and Microsoft" matches both AAPL and MSFT).

### D2: Timeseries Bucket Polling (FR-002/FR-004)

**Approach**: Extend `PollingService.poll()` to also fetch timeseries data. No separate class — the timeseries query depends on the ticker list derived from the sentiment items just fetched (same poll cycle).

**Query pattern**: After computing per-ticker aggregates, use the discovered ticker list to construct `BatchGetItem` requests for the timeseries table. For each ticker × resolution, compute PK = `{ticker}#{resolution}`, SK = `floor_to_bucket(now, resolution).isoformat()`.

**Ticker list derivation**: The SSE Lambda doesn't have `us-symbols.json`. Tickers are derived from the sentiment items already in memory. If a ticker has no sentiment items, it also has no timeseries records (timeseries is a write-time fanout from sentiment analysis).

**Optimization**: `BatchGetItem` fetches up to 100 items per request. With 22 tickers × 8 resolutions = 176 items, this requires 2 batch calls per poll cycle (~10ms total).

### D3: Event Wiring in Stream Generators

**Approach**: Both `generate_global_stream()` and `generate_config_stream()` use the same `poll_loop()` which returns raw data (metrics, per-ticker aggregates, timeseries buckets). Each generator maintains its own local state:
- `local_last_per_ticker: dict[str, TickerAggregate]` for sentiment change detection
- `local_last_buckets: dict[str, dict]` for timeseries change detection
- `local_is_baseline: bool` for first-poll suppression

Config streams apply `connection.matches_ticker()` before yielding `sentiment_update` and `partial_bucket` events.

**Why**: Per-connection snapshot state prevents the shared-state race condition. The `PollingService` remains stateless for per-ticker/bucket data (only `_last_metrics` stays shared for backwards compatibility with the existing `changed` flag on global metrics).

### D4: Event Buffer Sizing (FR-007)

**Approach**: Increase `EventBuffer.max_size` from 100 to 500 to accommodate the higher event volume (~200 events per poll cycle).

### D5: Baseline Establishment (FR-011)

**Approach**: Each generator coroutine initializes `local_is_baseline = True`. First poll populates local snapshots but does not yield change events. Subsequent polls diff against the snapshot. Since state is per-connection, each new connection independently establishes its own baseline.
