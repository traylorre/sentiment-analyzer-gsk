# Feature Specification: CORS 404 Handler

**Feature Branch**: `1311-cors-404-handler`
**Created**: 2026-04-02
**Status**: Draft
**Input**: When a request hits an unmapped route (e.g., `/api/v2/tickers/nonexistent-cors-test`), Powertools' `APIGatewayRestResolver` returns a default 404 WITHOUT CORS headers. The browser cannot read the response body and silently fails. Register a catch-all not-found handler that adds CORS headers to all unmatched-route 404 responses.

## Problem Statement

The sentiment analyzer dashboard uses Powertools `APIGatewayRestResolver` for routing. When a request arrives for a route that has no registered handler, Powertools returns a bare 404 response with no CORS headers. Because the `/{proxy+}` API Gateway route forwards ALL requests to the Lambda, these unmatched routes DO reach the Lambda -- it is Powertools, not API Gateway, that generates the headerless 404.

The existing `_make_not_found_response(origin)` function correctly builds 404 responses with conditional CORS headers, but it is only called by 13 specific environment-gated routes. Unmatched routes bypass it entirely.

The result: browsers receive an opaque CORS error instead of a readable 404, making debugging impossible for frontend developers and causing silent failures in production.

## Adversarial Review Findings

### Powertools `@app.not_found` Decorator

Powertools `APIGatewayRestResolver` supports a `@app.not_found` decorator that intercepts all unmatched routes before the default 404 is generated. This is the correct integration point -- it covers every unmatched route without modifying individual route handlers.

### Origin Header Availability

The `@app.not_found` handler has access to `app.current_event`, so `_get_request_origin()` works correctly within the handler context. No special plumbing needed.

### No API Gateway Gateway Response Needed

The `DEFAULT_4XX` gateway response in API Gateway would only cover 4xx errors generated by API Gateway itself (e.g., missing API key, throttling). Since the Lambda IS invoked for these requests (via `/{proxy+}`), the fix must be at the Lambda level. A `DEFAULT_4XX` gateway response would be defense-in-depth but is NOT required for this fix.

### Security: Origin Validation

The `_make_not_found_response(origin)` function already validates the origin against `_CORS_ALLOWED_ORIGINS` before adding CORS headers. Unknown origins get a bare 404 with only `Vary: Origin`. This is the correct security posture -- no changes needed.

### Idempotency with Existing Routes

The 13 environment-gated routes that already call `_make_not_found_response()` will continue to work unchanged. The `@app.not_found` handler only fires for routes with NO registered handler. Routes that ARE registered but return 404 internally (env-gated) are not affected.

## Functional Requirements

### FR-001: Catch-All Not-Found Handler

Register a Powertools `@app.not_found` handler that intercepts all unmatched routes and returns a 404 response with conditional CORS headers.

**Acceptance Criteria**:
- Requests to unmatched routes return HTTP 404 with `Content-Type: application/json`
- Response body is `{"detail": "Not found"}`
- When the `Origin` header matches an allowed CORS origin, the response includes `Access-Control-Allow-Origin`, `Access-Control-Allow-Credentials`, `Access-Control-Allow-Methods`, and `Access-Control-Allow-Headers`
- When the `Origin` header does not match an allowed origin, the response includes only `Vary: Origin` (no CORS headers)
- The `Vary: Origin` header is always present regardless of origin match

### FR-002: Logging

Log unmatched route requests at DEBUG level for operational visibility.

**Acceptance Criteria**:
- Each unmatched route hit produces a debug log entry with the request path and HTTP method
- Logging does not occur at INFO or higher to avoid log noise from scanners/bots

## User Scenarios

### US-001: Frontend Developer Hits Wrong Endpoint

A frontend developer accidentally requests `/api/v2/tickers/nonexistent`. The browser sends an `Origin` header matching the Amplify domain. The Lambda returns 404 with CORS headers. The developer sees a clear "Not found" error in their fetch response instead of an opaque CORS failure.

### US-002: Scanner Probes Random Paths

An automated scanner sends requests to `/admin`, `/wp-login.php`, etc. with no `Origin` header or an unknown origin. The Lambda returns 404 with `Vary: Origin` but no `Access-Control-Allow-Origin`. The scanner gets a standard 404 without information leakage about CORS configuration.

## Scope Boundaries

### In Scope

- Register `@app.not_found` handler in `handler.py`
- Reuse existing `_make_not_found_response()` and `_get_request_origin()`
- Unit test for the not-found handler

### Out of Scope

- API Gateway `DEFAULT_4XX` gateway response (defense-in-depth, separate feature)
- Modifying existing environment-gated routes
- OPTIONS preflight handling (already handled by API Gateway CORS config)
- Modifying `_make_not_found_response()` itself

## Dependencies

- `_make_not_found_response()` at handler.py:258-302 (existing, no changes)
- `_get_request_origin()` at handler.py:145-158 (existing, no changes)
- `_CORS_ALLOWED_ORIGINS` at handler.py:109-113 (existing, no changes)
- Powertools `@app.not_found` decorator support
