# Feature 1268: CORS Headers on Environment-Gated 404 Responses

## Problem Statement

Feature 1250 introduced environment gating for chaos endpoints and dashboard UI routes. When a request hits an env-gated route in non-dev environments (preprod, prod), the handler returns `_NOT_FOUND_RESPONSE` — a plain 404 with `{"detail": "Not found"}` and **no CORS headers**.

The response path is:
1. Browser sends cross-origin request to API Gateway
2. API Gateway forwards to Lambda via `AWS_PROXY` integration
3. Lambda handler hits env-gate check, returns 404 immediately
4. API Gateway passes Lambda response through AS-IS (no modification in `AWS_PROXY` mode)
5. Browser receives 404 **without** `Access-Control-Allow-Origin` header
6. Browser blocks the response (CORS violation) — frontend cannot read the 404 body

The error is opaque to the frontend. It cannot distinguish "endpoint doesn't exist" from "network error" or "CORS misconfiguration."

## Root Cause Analysis

### Why CORS headers are missing

The architecture has **three layers** that could add CORS:

1. **Lambda Function URL CORS config** (Terraform `function_url_cors` block): Configured, but the Function URL is now `AWS_IAM` auth-only (Feature 1256). Traffic routes through API Gateway, not directly to the Function URL. The Function URL CORS config only applies to requests that come directly through the Function URL.

2. **API Gateway Gateway Responses**: CORS headers are configured on `UNAUTHORIZED` (401), `MISSING_AUTHENTICATION_TOKEN` (401), and `ACCESS_DENIED` (403) gateway responses. There is **no** `DEFAULT_4XX` or `DEFAULT_5XX` gateway response. Gateway responses only fire for errors generated **by API Gateway itself** (auth failures, throttling), not for responses returned by the Lambda.

3. **Application-level CORS**: Intentionally absent. Tests at `test_dashboard_handler.py:585-626` verify the app does NOT add CORS headers, to avoid duplicate headers (documented architecture decision).

**The gap**: When the Lambda returns a 404 via `AWS_PROXY` integration, neither API Gateway nor the Lambda adds CORS headers. The response is returned to the browser bare.

### Scope of affected routes

Every route that uses `_NOT_FOUND_RESPONSE` in the env-gate pattern:
- `GET /` (dashboard index)
- `GET /favicon.ico`
- `GET /chaos` (chaos UI page)
- `GET /static/<filename>`
- `POST /chaos/experiments`
- `GET /chaos/experiments`
- `POST /chaos/experiments/<id>/start`
- `POST /chaos/experiments/<id>/stop`
- `POST /chaos/experiments/<id>/rollback`
- `GET /chaos/reports`
- `GET /chaos/reports/<id>`

These are all gated with `if not _is_dev_environment(): return _NOT_FOUND_RESPONSE`.

## Architecture Decision

### Option A: Add CORS headers to `_NOT_FOUND_RESPONSE` in application code

- **Pros**: Surgical fix, minimal blast radius, fixes all env-gated routes at once
- **Cons**: Breaks the "no application-level CORS" architecture decision. Creates potential for duplicate CORS headers if Lambda Function URL or API Gateway also adds them. Requires updating existing CORS tests.

### Option B: Add `DEFAULT_4XX` gateway response to API Gateway with CORS headers

- **Pros**: Keeps CORS at infrastructure level, consistent with existing architecture
- **Cons**: `DEFAULT_4XX` only applies to responses generated by API Gateway itself (quota exceeded, invalid API key, etc.), NOT to 4xx responses returned by the Lambda via `AWS_PROXY`. AWS documentation is explicit: "Gateway responses are applicable only when API Gateway fails to process an incoming request." Lambda 404s pass through unmodified.

### Option C: Add CORS headers to the `_NOT_FOUND_RESPONSE` specifically for env-gated responses

- **Pros**: Targeted fix only for the env-gated path. Does not change the general architecture principle. Clear documentation of WHY this response needs CORS headers (it's an early return before the resolver processes the request).
- **Cons**: Still technically application-level CORS, but with explicit justification.

### Decision: Option C

The env-gated 404 is a special case. It's an **early return** that bypasses the normal request processing pipeline. The architecture decision "no app-level CORS" was made to prevent **duplicate headers** when the infra layer (Function URL / API Gateway) also adds CORS. But in this case, neither infra layer adds CORS to Lambda-returned responses. The fix is targeted, documented, and doesn't create duplicate header risk because:

1. Lambda Function URL is AWS_IAM-only — direct browser access is impossible
2. API Gateway `AWS_PROXY` passes Lambda responses through AS-IS — no CORS injection
3. API Gateway Gateway Responses only fire on API Gateway errors, not Lambda returns

## Functional Requirements

### FR-001: CORS headers on env-gated 404 responses

The `_NOT_FOUND_RESPONSE` returned by environment gating MUST include CORS headers that allow the browser to read the response body.

Required headers:
- `Access-Control-Allow-Origin`: Must reflect the requesting origin if it matches allowed origins. Use the `CORS_ORIGINS` environment variable (already available in Lambda env, set from `var.cors_allowed_origins`).
- `Access-Control-Allow-Credentials`: `true` (consistent with existing CORS config)
- `Access-Control-Allow-Methods`: `GET,POST,PUT,DELETE,PATCH,OPTIONS` (consistent)
- `Access-Control-Allow-Headers`: `Content-Type,Authorization,Accept,Cache-Control,Last-Event-ID,X-Amzn-Trace-Id,X-User-ID` (consistent)

### FR-002: Origin validation on env-gated 404

The CORS origin header MUST NOT be a wildcard (`*`). It must validate the requesting `Origin` header against the configured allowed origins list (from `CORS_ORIGINS` env var). If the origin is not in the allowed list, omit `Access-Control-Allow-Origin` entirely (fail-closed).

### FR-003: Consistent CORS across all env-gated responses

ALL routes that use the env-gating pattern (`if not _is_dev_environment(): return ...`) MUST use the same CORS-enabled 404 response.

### FR-004: No CORS changes to non-gated responses

Responses that flow through the normal Powertools resolver pipeline MUST NOT have application-level CORS headers added. The existing architecture decision (infra-level CORS) remains unchanged for all non-gated paths.

### FR-005: Preflight handling for env-gated routes

OPTIONS requests to env-gated routes are handled by API Gateway MOCK integrations (already configured). No application-level change needed for preflight. This FR is a verification requirement — confirm that OPTIONS requests already work correctly in non-dev environments.

## Non-Functional Requirements

### NFR-001: Performance

CORS header addition MUST NOT add measurable latency. The origin validation is a simple set-membership check against a small list of origins (typically 2-5 entries).

### NFR-002: Security

- No wildcard origins (FR-002)
- Origin validation must be fail-closed (no CORS headers if origin unknown)
- Must not leak environment information beyond "Not found"
- `Access-Control-Allow-Credentials: true` MUST NOT be used with wildcard origin (per CORS spec, this combination is invalid and browsers reject it)

### NFR-003: Observability

Log when an env-gated 404 is returned with CORS headers, including the validated origin (if any), at DEBUG level. Do not log at INFO/WARN to avoid log noise in production.

## Design Constraints

1. The `_NOT_FOUND_RESPONSE` is a module-level constant. Since CORS requires checking the request's `Origin` header, the response cannot be a static constant — it must be generated per-request. Convert to a function: `_make_not_found_response(origin: str | None) -> Response`.

2. The `CORS_ORIGINS` environment variable contains a comma-separated list of allowed origins. Parse this at module level (same pattern as `ENVIRONMENT`).

3. All 12 env-gated routes must be updated to call the function instead of referencing the constant.

## Test Requirements

- Test that `_make_not_found_response` with valid origin includes CORS headers
- Test that `_make_not_found_response` with invalid/missing origin omits `Access-Control-Allow-Origin`
- Test that `_make_not_found_response` returns 404 with correct body
- Test that env-gated routes in preprod/prod return 404 WITH CORS headers when Origin is valid
- Test that env-gated routes in preprod/prod return 404 WITHOUT CORS origin when Origin is invalid

### Integration Tests
- Test that chaos endpoints through LocalStack API Gateway return 404 with CORS headers in non-dev

### E2E Tests
- Test that chaos endpoints in preprod return 404 with CORS headers readable by browser
- **Open question**: E2E tests require a deployed preprod environment. May need to be deferred to gameday validation.

### Playwright Tests
- Test that frontend correctly receives and displays 404 from env-gated endpoints
- Test that frontend error handling works for chaos page in non-dev environment
- **Open question**: Playwright tests need a running frontend + backend. The chaos.html page is itself env-gated, so testing frontend behavior when the page itself is 404'd may not be meaningful. The real test is: API calls from a frontend running in dev that hits a non-dev backend. This is an unusual topology.

## Open Questions

1. **Playwright test topology**: The chaos.html page is itself env-gated. If the frontend is in dev but the backend is in preprod, the chaos page loads (from dev) but API calls fail (to preprod). Is this a realistic scenario? If not, Playwright tests may only need to verify error handling for API call failures, not page load failures.

2. **Static asset env gating**: Routes like `GET /`, `GET /static/<filename>`, `GET /favicon.ico` are also env-gated. These serve HTML/CSS/JS, not JSON APIs. Browsers don't make CORS preflight for same-origin navigation requests. CORS headers on static asset 404s are only needed if the frontend tries to fetch them via JavaScript (e.g., `fetch('/static/app.js')`). Clarify if this is needed.

3. **Origin header presence**: Not all HTTP clients send the `Origin` header. Curl, Postman, and server-to-server requests typically omit it. The CORS spec says: if no `Origin` header, don't add CORS response headers. This is correct behavior — non-browser clients don't need CORS.

### Finding AR1-001: `CORS_ORIGINS` env var exists but is unused in handler (CONFIRMED)

**Severity**: Informational (helps implementation)

The spec correctly identifies `CORS_ORIGINS` as the source for allowed origins. The implementation must add `CORS_ORIGINS = os.environ.get("CORS_ORIGINS", "")` and parse it.

**Resolution**: Spec is correct. No change needed — this is a design constraint that's already documented.

### Finding AR1-002: _NOT_FOUND_RESPONSE is duplicated at line 123 AND line 226 (BUG)

**Severity**: Medium

The handler defines `_NOT_FOUND_RESPONSE` TWICE:
- Line 123-127 (first definition, never used — it's in the pre-refactor position)
- Line 226-230 (second definition, actually used by all env-gated routes)

The first definition is dead code. The spec must note this — the fix should remove the dead definition at line 123 and only modify the one at line 226 (or rather, replace it with the function).

**Resolution**: Added to implementation note. Remove the dead `_NOT_FOUND_RESPONSE` at line 123-127.

### Finding AR1-003: Origin extraction from Powertools event needs verification

**Severity**: Medium

The spec says "check the request's `Origin` header." But inside a Powertools route handler, how do you access headers? The handler uses `app.current_event.raw_event` in some places and `app.current_event` in others. The `Origin` header will be in `app.current_event.headers` (Powertools normalizes headers to lowercase). But the `_make_not_found_response` function is called INSIDE route handlers, so `app.current_event` is available.

However, the spec proposes `_make_not_found_response(origin: str | None)` as the function signature. This means each call site must extract the origin and pass it. Alternative: the function could read `app.current_event.headers.get("origin")` directly — but this couples it to the Powertools resolver context.

**Resolution**: Keep the explicit parameter approach. It's more testable (no dependency on global resolver state) and makes the origin extraction explicit at each call site. But add a helper to reduce boilerplate:

```python
def _get_request_origin() -> str | None:
    """Extract Origin header from current request."""
    try:
        return app.current_event.headers.get("origin")
    except Exception:
        return None
```

Then call sites become: `return _make_not_found_response(_get_request_origin())`

### Finding AR1-004: Spec doesn't address Vary header (SPEC GAP)

**Severity**: Low

Per HTTP caching spec, when a response varies based on the `Origin` request header, it should include `Vary: Origin`. Without this, a CDN/proxy could cache a response with CORS headers for origin A and serve it to origin B. In practice, this is low risk because:
1. 404 responses are not typically cached
2. There's no CDN/cache in front of API Gateway for this path

But for correctness, add `Vary: Origin` to the response headers.

**Resolution**: Add `Vary: Origin` to the CORS response headers.

### Finding AR1-005: Spec incorrectly claims "early return before resolver processes it"

**Severity**: Medium (spec accuracy)

The spec says in the Decision section: "It's an early return that bypasses the normal request processing pipeline." This is WRONG. The env-gated `return _NOT_FOUND_RESPONSE` happens INSIDE a Powertools route handler function (e.g., `@app.post("/chaos/experiments")`). The resolver DID route the request to the correct handler. The `Response` object is returned from the handler and serialized by Powertools into the Lambda response dict via `app.resolve()`.

This matters because it means the response DOES flow through Powertools serialization. If Powertools had CORSConfig, it would add CORS headers. Since it doesn't (architecture decision), no CORS headers are added.

The fix is still correct (add CORS headers to the Response object's `headers` parameter), but the justification needs correcting.

**Resolution**: Correct the Decision section language. The env-gated 404 is not an "early return before the resolver" — it's a route handler returning a custom Response without CORS headers. The fix adds CORS headers to this specific Response.

### Finding AR1-006: Test helper `_reload_handler_with_env` doesn't set CORS_ORIGINS

**Severity**: Medium (test gap)

The existing `_reload_handler_with_env("preprod")` only patches `ENVIRONMENT`. The new implementation will also need `CORS_ORIGINS` set for the origin validation to work. Tests must patch both env vars.

**Resolution**: Test plan must ensure `CORS_ORIGINS` is set when reloading the handler for CORS tests.

### Finding AR1-007: 12 call sites is fragile — consider decorator pattern

**Severity**: Low (implementation suggestion)

Updating 12 individual `if not _is_dev_environment(): return _NOT_FOUND_RESPONSE` blocks is error-prone. Future routes could forget the pattern. Consider a decorator:

```python
def env_gated(fn):
    @functools.wraps(fn)
    def wrapper(*args, **kwargs):
        if not _is_dev_environment():
            return _make_not_found_response(_get_request_origin())
        return fn(*args, **kwargs)
    return wrapper
```

However, this is a refactoring improvement, not a bug fix. The feature scope should stay focused on the CORS fix. The decorator could be proposed as follow-up tech debt.

**Resolution**: Note as future improvement in plan.md. Keep the feature scope to the CORS fix.

### Finding AR1-008: Open Question #2 resolution (static asset CORS)

**Severity**: Low

Self-resolving: Static asset routes (`/`, `/static/<filename>`, `/favicon.ico`) are served via JavaScript `fetch()` only if the frontend dynamically loads them. In this architecture, the main dashboard page is an Amplify/Next.js frontend that talks to the API. The static routes are for the HTMX internal dashboard which loads in the same origin. Cross-origin fetch of static assets is unlikely but harmless to protect. Since all these routes use the same `_NOT_FOUND_RESPONSE`, the fix applies uniformly. No special case needed.

**Resolution**: Open Question #2 resolved. All env-gated routes get CORS headers uniformly. No special handling for static vs API routes.

## Clarification (Stage 4)

### CQ-001: How does Powertools serialize Response headers into the Lambda response dict?

**Investigation**: The Powertools `Response` class accepts a `headers` parameter of type `Mapping[str, str | list[str]]`. When `app.resolve(event, context)` processes a route handler's return value, it serializes the Response into a dict with `statusCode`, `headers`, `body`, etc. Custom headers from the Response are merged into the response dict's `headers` field. API Gateway `AWS_PROXY` integration passes these through to the client.

**Confirmed**: Adding `headers={"Access-Control-Allow-Origin": origin}` to the `Response` constructor WILL result in the header being present in the final HTTP response to the browser.

### CQ-002: Does the `_NOT_FOUND_RESPONSE` at line 123-127 cause any side effects if removed?

**Investigation**: The first `_NOT_FOUND_RESPONSE` definition at line 123-127 is shadowed by the second definition at line 226-230 (Python uses the last assignment). No code between line 127 and line 226 references `_NOT_FOUND_RESPONSE`. The first definition is dead code — removing it has zero behavioral impact.

**Confirmed**: Safe to remove.

### CQ-003: Where should Playwright tests go?

**Investigation**: Existing Playwright tests are in `frontend/tests/e2e/*.spec.ts`. The Playwright config is at `frontend/playwright.config.ts`. Tests use `@playwright/test` and follow TypeScript conventions. Chaos-specific tests already exist (chaos.spec.ts, chaos-error-boundary.spec.ts, etc.).

**Resolution**: New Playwright test should go in `frontend/tests/e2e/cors-env-gated-404.spec.ts`.

### CQ-004: What happens with `Access-Control-Allow-Credentials: true` and origin validation?

**Investigation**: Per CORS spec RFC 6454, when `Access-Control-Allow-Credentials: true` is set:
- `Access-Control-Allow-Origin` MUST be a specific origin (not `*`)
- `Access-Control-Allow-Headers` MUST be a specific list (not `*`)
- `Access-Control-Allow-Methods` MUST be a specific list (not `*`)

Our implementation never uses `*` — it reflects the validated origin. This is correct.

### CQ-005: Will the existing CORS tests at test_dashboard_handler.py:585-626 break?

**Investigation**: The test `test_cors_handled_by_lambda_function_url` (line 585) just reloads the module and checks for log output — no assertions about CORS headers. The test `test_no_cors_middleware_in_app` (line 610) only asserts `app is not None`. Neither test will break from adding CORS headers to `_make_not_found_response()`.

However, the DOCSTRINGS in these tests say "CORS is now handled at the infrastructure level" and "Lambda Function URL handles CORS exclusively." These need updating to note the exception for env-gated responses.

**Resolution**: Update docstrings in the two CORS tests to document the env-gated exception. No assertion changes needed.

### CQ-006: Is there a test helper for setting `CORS_ORIGINS` in `_reload_handler_with_env`?

**Investigation**: The current helper only patches `ENVIRONMENT`. Need to extend it (or create a new helper) that patches both `ENVIRONMENT` and `CORS_ORIGINS`. The cleanest approach: don't modify the shared helper, instead use `patch.dict(os.environ, {...})` directly in the new CORS test class, setting both variables.

**Resolution**: New CORS tests use their own reload helper or explicit `patch.dict`.
